# 候选池与观察池规则

## 核心原则

这个市场变化太快，默认不应该依赖静态观察池做推荐。

更科学的流程是：

```text
每次扫描 -> 先拉取最新市场数据 -> 动态生成候选池 -> 深度评分 -> 输出报告
```

`config/watchlist.csv` 只作为“必须复查列表”，不能作为机会来源的主体。

## 当前观察池是什么

`config/watchlist.csv` 是手工种子池，不是全市场扫描结果。

它的作用：

- 快速验证扫描器、评分模型和日报格式。
- 保存用户已经感兴趣或经常复看的标的。
- 在 API 配额有限时作为保守候选集。
- 保存持仓和想重点盯住的标的，确保它们不会被 discovery 漏掉。

它不能代表：

- 全市场最优机会。
- 当天最强爆发品。
- 所有符合 T+7 的候选。

因此，只基于 `watchlist.csv` 输出的报告必须被理解为“观察池扫描”，不能写成“全市场推荐”。

## 防止观察池误导

系统必须区分三种扫描范围：

- `watchlist`：只扫本地观察池，适合跟踪熟悉标的。
- `discovery`：每次用 CSQAQ 排行榜、列表、价格数据重新生成候选池，再交给 SteamDT 验证。
- `full_market`：全市场扫描。只有当 API 权限、配额和数据完整性足够时才能这样标记。

默认优先级：

```text
discovery > watchlist
```

也就是说：推荐来源默认应该是新拉取的 discovery 结果；观察池只用于补充复查，不用于限制机会范围。

如果扫描范围不是 `full_market`，报告必须写明：

- 本次不是全市场最优。
- 候选池来源。
- 可能漏掉观察池外的机会。

## 推荐更新频率

观察池分三层更新：

- 每次推荐前：运行 `discovery`，重新拉取最新候选。
- 每日：运行 `watchlist` 扫描，跟踪已有候选和持仓，但不把它当作全市场推荐。
- 每月：复盘并清理观察池，删除长期无流动性、长期跑输板块、长期规则不命中的标的。

如果市场出现重大事件，比如版本更新、赛事、饰品大盘急跌或急涨，应临时运行一次 `discovery`。

## 候选发现流程

第一层：CSQAQ 粗筛

- 排除箱子、收藏包、胶囊。
- 优先普通皮肤、主战皮肤、手枪皮肤、高流动性红/粉皮。
- 按价格上限、7 日涨幅、30 日涨幅、在售数量、求购数量、板块强弱筛选。
- 保留 50-200 个候选。

第二层：SteamDT 验证

- 用 `price/batch` 查平台最低价、最高求购价、卖单和买盘深度。
- 用 `item/v1/kline` 查 7 日和 30 日走势。
- 计算 T+7 分数。

第三层：加入观察池

只有满足以下条件的标的才进入 `config/watchlist.csv`：

- 不属于默认排除范围。
- 有足够流动性。
- 点差没有明显异常。
- 至少有一个清楚的交易理由。
- 能写出无效条件。

## 为什么不完全删除观察池

观察池不适合做“发现机会”的主入口，但仍然有价值：

- 持仓必须每天复查，即使它当天不在涨幅榜。
- 一些稳健票不会每天出现在热门榜，但适合持续跟踪。
- API 配额或服务异常时，需要一个最低限度的复查列表。
- 复盘时需要知道某个标的为什么被长期观察、为什么被移除。

因此正确关系是：

```text
fresh discovery 负责找机会
watchlist 负责跟踪和复盘
holdings 负责风险管理
```

## 报告措辞规则

当扫描范围是观察池时：

- 可以说“观察池里当前排名靠前的是...”。
- 不可以说“今天全市场最值得买的是...”。

当扫描范围是发现扫描时：

- 可以说“在本次发现池中排名靠前的是...”。
- 不可以说“全市场最优”，除非确实完成全市场扫描。
